# CLAUDE.md
이 파일은 Claude Code가 이 프로젝트에서 작업할 때 따라야 하는 지침입니다.

## 대화 방식

- 기본적으로 한국어로 답변합니다.
- 사용자는 안드로이드 앱 개발자로 웹 개발은 처음입니다. 안드로이드/Java/Kotlin 개념에 빗대어 설명하면 이해하기 쉽습니다.
- 어려운 웹 개발 용어는 가능한 한 피합니다.
- 꼭 필요한 개발 용어를 써야 할 때는, 그 단어가 무슨 뜻인지 짧게 풀어서 설명합니다.
- 간결함보다 이해하기 쉬운 설명을 우선시합니다. 짧게 줄이지 말고 풀어서 설명합니다.
- 설명의 분량이나 상세함을 줄이지 않습니다. 내용은 그대로 유지하되, 쉬운 말로 풀어서 설명합니다.
- 코드 흐름을 설명할 때는 "1단계 → 2단계 → 3단계" 식으로 시간 순서대로 풀어서 설명합니다.
- 간단하고 유지보수하기 쉬운 솔루션을 우선시합니다.

## 응답 수준 가이드

질문 종류에 따라 검증 수준을 다르게 적용합니다:

- **단순 질문** (값 확인, 함수 위치 등): Read 1~2회로 즉시 답변. 전체 흐름 추적 불필요
  - ⚠️ **예외**: 질문에 특정 파일명/함수명/변수명이 구체적으로 언급되면, 아는 것 같아도 반드시 Read로 실물 확인. 기억·추측으로 답하지 말 것
- **일반 질문** (동작 흐름, 기능 설명 등): 관련 파일만 읽고 답변
- **오류/버그 분석**: "오류 분석 시 필수 절차" 전체 수행
- **코드 수정 작업**: "코드 수정 원칙" 및 "코드 수정 전 필수 절차" 전체 수행

## MVVM 설명 규칙

코드를 수정하거나 추가할 때, 반드시 다음 두 가지를 설명합니다:

1. **MVVM 레이어**: 이 코드가 Model / ViewModel / View 중 어디에 해당하는지
2. **파일 선택 이유**: 왜 이 파일에 넣었는지 (이 파일이 어떤 역할을 하는 파일인지)

이 프로젝트의 MVVM 구조는 다음과 같습니다:

| MVVM 레이어 | 이 프로젝트의 파일 | 안드로이드 비유 |
|---|---|---|
| View | `src/views/*.vue`, `src/components/*.vue`의 `<template>` 영역 | XML 레이아웃, Activity/Fragment |
| ViewModel | `src/composables/use***.js` | ViewModel 클래스 |
| Model | `src/api.js`, `src/board.js` | Repository, 데이터 클래스 |

## 코드 수정 방식

- 코드를 수정하거나 기능을 추가할 때, 관련 코드 전체에 주석을 추가합니다.
- 주석은 신입사원(웹 개발 입문자)도 이해할 수 있게 자세히 적습니다.
- 주석에는 다음을 포함합니다:
  - 이 코드가 어떤 기능을 하는지
  - 왜 이렇게 작성했는지
  - 사용된 문법이나 함수가 무엇인지 (예: `includes()`, `sort()`, 삼항연산자 등)
  - 가능하면 안드로이드 개념과 비교 (예: `ref()` → LiveData, `router.push()` → startActivity())
- 주석을 달 때는 해당 파일에서 쓰는 주석 문법도 간단히 설명합니다.
  - HTML: `<!-- 설명 -->`
  - CSS: `/* 설명 */`
  - JavaScript/Vue: `// 설명` 또는 `/* 설명 */`
- 화면에 보이는 부분을 바꾼 경우, 사용자가 화면에서 무엇을 다르게 보게 되는지도 설명합니다.
- 서버 동작을 바꾼 경우, 어떤 요청/파일/데이터 흐름이 영향을 받는지 설명합니다.

## 프로젝트 구조

## 작업 안전 규칙

- 사용자가 직접 바꾼 내용을 함부로 삭제하거나 되돌리지 않습니다.
- 코드를 바꾸기 전에는 주변 코드를 먼저 읽고, 기존 방식에 맞춰 수정합니다.
- 큰 구조를 한 번에 바꾸기보다, 필요한 부분만 작고 안전하게 수정합니다.

## 정직성 규칙 (할루시네이션 방지)

- 질문에 **특정 파일명 / 함수명 / 변수명**이 언급되면, 아는 것 같아도 반드시 Read 도구로 실물을 확인한 후 답변합니다. 기억이나 추측으로 답하지 않습니다.
  - 예: "`usePostList.js`에서 검색 로직이 어떻게 돼 있어?" → 반드시 파일을 열어서 확인. 기억으로 답하지 말 것
- 확인이 불가능하면 "확인 필요" 또는 "모르겠습니다"라고 솔직히 답합니다.
- "모르겠습니다", "확인 필요"는 잘못된 답보다 훨씬 가치 있는 정답입니다.

## 답변 전 자가 검증

답변하기 전에 스스로에게 확인합니다:

1. 이 답변의 근거가 되는 코드를 Read로 읽었는가?
2. 파일명:라인번호를 명시할 수 있는가?
3. 못 한다면 "추측" 또는 "확인 필요"로 표시했는가?

하나라도 No이면 Read 도구로 확인한 후 답변합니다.

## 코드 수정 원칙

- 기존 코드를 삭제할 때는 반드시 삭제 이유를 설명하고, 사용자에게 확인받은 후에만 삭제합니다.
- boolean 변수(true/false 플래그)를 이용한 임시방편 해결보다 근본적인 원인을 찾아 해결합니다.
- 새로운 변수나 함수를 추가하기보다 기존 코드를 정리하는 방향을 우선시합니다.
- 검증 없이 코드를 원복하거나 되돌리지 않습니다.
- 수정 후 반드시 기존 기능이 정상 동작하는지 검수합니다.

## 대안/해결책 제시 체크리스트

코드를 제안하기 전에 반드시 확인합니다:

1. **현재 코드와 비교**
   - 현재 코드가 있으면 먼저 분석하고 장점을 파악합니다.
   - 제안하는 코드가 **구체적으로 어떤 점이 더 나은지** 비교합니다.
   - 비교 없이 대안을 제시하지 않습니다.

2. **실제 적용 시 문제점 검토**
   - 지연(delay)이 발생하는가?
   - 부작용(side effect)은 없는가?
   - 사용자 경험에 악영향은 없는가?

3. **자문(自問)**
   - "이 코드를 실제로 쓸 것인가?" → No라면 제시하지 않습니다.
   - "현재 코드보다 나은가?" → No라면 제시하지 않습니다.
   - "사용자가 요청한 것인가?" → No라면 제시하지 않습니다.
   - 현재 코드가 최선이라고 판단되면, 대안 없이 현재 코드를 설명합니다.

## 코드 수정 전 필수 절차

기존 기능을 수정하거나 새 기능을 추가할 때 반드시 아래 순서를 따릅니다:

### 1단계: 기존 코드 동작 흐름 분석
- 수정 대상 함수/컴포넌트가 호출되는 모든 경로 파악
- 관련된 다른 파일(api.js, composables, server.js 등)의 동작도 함께 파악
- **"이 코드가 현재 어떻게 동작하는지"를 사용자에게 먼저 설명합니다**

### 2단계: 수정 계획 설명 및 승인
- 수정이 필요한 이유(원인) 설명
- 구체적인 수정 내용 설명 (어떤 파일의 어떤 코드를 어떻게 바꿀 것인지)
- 기존 기능에 영향을 줄 수 있는 부분 명시
- **"수정하시겠습니까?" 확인 후 승인받은 후에만 수정합니다**

### 3단계: 수정 후 검증
- 수정한 코드가 기존 기능을 깨뜨리지 않았는지 확인
- 문제 발생 시 즉시 원복하고 원인 재분석

**하지 말아야 할 것:**
- 기존 코드를 충분히 읽지 않고 새 로직 추가
- 사용자 승인 없이 일방적으로 코드 수정
- 사용자가 요청하지 않은 대안을 여러 개 나열하기

## 오류 분석 시 필수 절차

버그/오류 질문을 받으면 반드시 아래 순서대로 진행합니다:

1. **사용자의 모든 단서 메모**
   - "버튼 누르면 발생" → 중복 호출 가능성
   - "새로고침하면 사라짐" → 상태(state) 초기화 문제 가능성
   - "특정 게시판에서만 발생" → 데이터 조건 문제 가능성
   - 재현 조건, 타이밍, 빈도 등 모든 단서 기록

2. **Grep으로 전체 검색**
   - 오류와 관련된 함수명/변수명/API 경로를 프로젝트 전체에서 검색
   - 해당 코드가 사용되는 **모든 위치** 확인

3. **Read로 관련 파일 전체 읽기**
   - 부분이 아닌 **전체 로직** 파악
   - 관련된 View(.vue), ViewModel(composables), Model(api.js, server.js) 모두 확인

4. **실행 흐름 시뮬레이션**
   - "이 상황에서 어떤 순서로 코드가 실행되는가?"
   - "비동기(async/await) 타이밍 문제는 없는가?"
   - "여러 곳에서 동시에 호출될 가능성은 없는가?"

5. **확신이 생긴 후에만 답변**
   - 위 1~4단계를 완료했는가?
   - "~일 것 같습니다", "아마도~" 같은 추측성 답변 금지
   - 불확실하면 "전체 흐름을 먼저 확인하겠습니다"라고 솔직히 말하기

## 코드 변경 검증 원칙

- 코드 수정 후 문제가 발생했다는 피드백이 있으면, 반드시 `git diff`로 이전 코드와 현재 코드를 비교합니다.
- "원래부터 있던 문제"라고 답하기 전에, 반드시 이전 커밋의 해당 코드를 확인합니다.
- 현재 코드만 보고 "이게 원래 코드"라고 추측하지 않습니다.
- 사용자가 확인을 요청하면, 실제 동작 흐름을 끝까지 추적합니다. 전체 실행 흐름을 따라가며 검증합니다.
